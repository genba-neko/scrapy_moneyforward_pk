### T2: spider URL の variant 駆動置換 (Issue B)

**Priority**: P1　**工数**: M (4-6h)　**回復見込**: +6 (機能移植率 +3, 設計拡張性 +2, スパイダー正確性 +1)　**依存**: T1

**問題**:
- `src/moneyforward_pk/spiders/transaction.py:20` `TRANSACTION_URL = "https://moneyforward.com/cf"` 等、3 spider すべてが URL ハードコード。
- registry が `VariantConfig.transactions_url` / `accounts_url` / `asset_allocation_url` を提供しているのに、本体 spider が無視。「設計が空証明」状態 (c3 iter1 補正採点 §3 設計拡張性 -3)。
- xmf_ssnb_transaction を起動すると `MfTransactionSpider._month_request` が module 定数 `TRANSACTION_URL` を参照するため **mf 本体 URL に飛ぶ非機能スパイダー**。

**実装方針**:
1. `MoneyforwardBase` に `variant_name: str = "mf"` クラス属性 + `__init__` で `self.variant = get_variant(self.variant_name)` を解決。`start_url` / `is_partner_portal` を `self.variant.*` から自動代入。
2. 3 spider 本体を変更:
   - `MfTransactionSpider`: module 定数 `TRANSACTION_URL` 削除、`_month_request` 内で `self.variant.transactions_url` 参照。`allowed_domains` を `urlparse(self.variant.base_url).netloc` から動的算出 (`from_crawler` 内)。
   - `MfAccountSpider`: module 定数 `ACCOUNTS_URL` 削除、`_accounts_request` 内で `self.variant.accounts_url` 参照、allowed_domains 同様。
   - `MfAssetAllocationSpider`: module 定数 `PORTFOLIO_URL` 削除、`after_login` 内で `self.variant.asset_allocation_url` 参照、allowed_domains 同様。
3. `XMoneyforwardLoginMixin` の login_flow に variant の `login_form_email` / `login_form_password` を反映 (現状: hardcoded `sign_in_session_service[email]`)。`MoneyforwardBase.login_flow` も同様に variant の form name を使うよう変更。これで `mf` (`mfid_user[email]`) と `xmf_ssnb` (`sign_in_session_service[email]`) を 1 メソッドで対応可能。
4. 既存 mf 系 3 spider のテストで挙動が完全に従来と一致することを assert (URL/form name の互換性テスト追加)。
