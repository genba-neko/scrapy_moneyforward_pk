# CLAUDE.md
## プロジェクトルール

## 対話・禁止事項

- 推測に基づく回答・行動禁止
- 敬意と畏怖をもってユーザに接し、丁寧な言葉使いで対応すること
- 指摘に対して事実確認を行い対応すること
- ごまかし、うわべだけの対応禁止
- masterへのコミット・pushの禁止
- PR 未作成のまま merge 禁止
- squash merge 禁止（merge commit を使う）

### プラン作成

- 保存先: `plan/YYYYMMDD_HHMM_概要.md`（時刻: `date +%Y%m%d_%H%M`）
- 開始時: 【プラン作成モード開始】を応答

### issue 登録

- ユーザー価値・機能単位で分割。細かい準備/docs 追従は本体 issue に吸収
- issue 本文に「関連資料」セクションで `plan/` ファイルへリンク
- ラベル: `feat` / `fix` / `refactor` / `test` / `chore` / `docs`
- issue粒度を勝手に決めて乱立させることを禁止する（複数の場合は要ユーザ確認）

### 実装フロー

1. ブランチ名提案 → 決定: 人間
2. 設計・実装・レビュー・テスト
3. コミット: 人間判断
4. PR: 人間判断
5. **プラン完了マーク**: コミット前に `plan/` 該当 issue へ `[完了 PR#XX YYYY-MM-DD]` 追記
6. PRマージ: 人間判断

- プラン議論中は実装着手しない（コード diff・新規ファイル生成禁止）
- 実装フェーズはユーザー明示指示後のみ開始
- `commit` / `push` / `merge` / `rebase` / PR 作成はユーザー明示指示まで実行しない
- 実装完了後: push → PR → PR番号を使いプランに完了マーク → push → master マージ → `git pull`
- **プランと実態が乖離したら即時 `plan/` 更新（確認不要）**
- **他エージェントへの依頼は人間の明示指示時のみ**

### ブランチ命名

`feature/[issue番号]_[概要]` / `fix/[issue番号]_[概要]`

### コミット規則

Conventional Commits: `feat:` / `fix:` / `chore:` など。

### 複数環境運用ルール (PC / NAS)

- **publish / r2_sync は NAS 専用**: index.json は R2 の実態ではなくローカル履歴から生成するため、
  runtime/ が空の環境から publish すると R2 index が切り詰まり consumer が過去データを参照できなくなる。
- **PC での orchestrate 実行**: 必ず `-SkipPublish` フラグを付ける。フラグなし実行は禁止。
  → r2_sync に縮小検出ガードあり (abort + 警告) だが、それは最後の安全網であり運用で防ぐのが本筋。
- **PC でのデバッグクロール**: `-SkipPull` モード (crawler のみ、publish なし) を使う。

### 破壊的操作の鉄則

削除・上書き・切り捨てを含む設計はコード前に人間へ危険性を説明し、安全側を提案。

1. **事前説明**: 消失リスクに気づいたらコード前に伝える
2. **安全側に倒す**: 削除より移動・アーカイブ。上書き前にバックアップ。`-Force` より `-WhatIf`
3. **2ステップ順序保証**: 保存成功しない限り削除しない設計
